Documentation

orderbook is a Go library for running a central limit order book (CLOB) and matching engine — the core of an exchange, a trading simulator, or any system where buyers and sellers meet at a price. It is embeddable, uses exact decimals for money (never floats), is deterministic, and ships the order types real venues use.

go get github.com/intrepidkarthi/orderbook/pkg/matching@v0.1.0

New to order books? Start with the from-scratch guide. For the full API, see pkg.go.dev.

Architecture

The project follows one rule: strict downward layering. The core library depends on nothing above it; research, simulation, strategy, surveillance, and the web demo build on top of it. This keeps the core lean and importable on its own.

apps / web / examples         (consume everything below)
backtest · strategy · sim
signals · marketdata
surveillance · auction
─────────── CORE ───────────
matching   (engine)
orderbook  (book + market data)
types      (Order, Trade, errors)

Dependencies flow one direction only — downward. You import just the layers you need; most consumers use only pkg/types, pkg/orderbook, and pkg/matching.

Order book internals

Data structure

Each side of the book is a map[price]→level for O(1) level lookup, plus a price-sorted slice maintained by binary-search insertion. Reading the best price is O(1); inserting a brand-new price level is O(log n) to find its slot. Each level is a FIFO queue of orders — which is what gives time priority.

Price–time priority

An incoming order matches against the best price first; among orders at the same price, first-come-first-served (the FIFO queue). Trades always print at the resting (maker) order's price.

Exact-decimal money

Prices and quantities are shopspring/decimal, never floats. Binary floating point silently accumulates rounding error on money; this engine does not. It is the single most important correctness decision, and the main allocation cost on the hot path — an int64 fixed-point fast path behind the same interface is a documented future optimization.

Determinism

Given the same ordered input stream, the engine produces byte-identical trades and book state. There is no wall-clock or RNG in the matching path; anything that must be reproducible (timestamps, ids) is injected. This is what makes record→replay and golden-file tests possible — a recorded session replays to an identical trade digest.

Concurrency

One writer per book, guarded by a mutex; readers take snapshots. A single book is inherently sequential at the match point, so you scale out across symbols (one book/goroutine per symbol), not by parallelizing one book.

Order types

TypeBehavior
MarketTakes liquidity immediately; may sweep several levels.
LimitRests at a price or better; can be a maker.
Stop / Stop-limitDormant until the market reaches a trigger price, then fires as a market/limit order.
IcebergShows only a display slice; the hidden reserve auto-refills as it fills.

PeggedPrice derived from a reference (bid / ask / mid) plus a signed offset.
OCO / bracketTwo linked orders; one completing cancels the other.
Trailing stopA stop that ratchets with the market by a fixed trail and never retreats.

Time-in-force: GTC (rest until cancelled), IOC (fill now, cancel the rest), FOK (fill entirely now or cancel all).

Matching

An incoming order crosses the resting book by price–time priority until it is filled or the book stops crossing. The resting side is the maker; the crossing order is the taker. Trades print at the maker's price, so a large taker "walks the book" and pays a worse average price (slippage).

Self-trade prevention

When an order would match the same user's resting order, one of four modes applies: cancel-newest (default), cancel-oldest, cancel-both, or allow.

Pro-rata

An optional per-engine mode. At each price level, an incoming order's quantity is allocated in proportion to resting size instead of strictly FIFO, with the rounding remainder distributed so allocations sum exactly.

Auctions

A call-auction uncross computes the single clearing price that maximizes matched volume across resting bids and asks — the mechanism behind opening and closing auctions, where orders accumulate and cross once at a common price.

Market data

Three granularities of book view:

  • L1 — top of book (best bid/ask and sizes).
  • L2 — aggregated size per price level.
  • L3 / MBO — every resting order individually, price-then-time ordered.

Snapshots carry a monotonic sequence number for gap detection, and executions are available as a trade tape (time & sales).

Market integrity

Controls that keep a venue orderly, in the core and in a surveillance layer above it:

  • Circuit breakers — a price-band collar rejects limit orders priced beyond ±band from the last trade; trading can be halted and resumed.
  • Stop-cascade protection — triggered stops chain in a bounded cascade rather than an unbounded loop.
  • Surveillance — spoofing/layering detection (large orders cancelled unfilled soon after placement) and order-rate limits (quote-stuffing).

Getting started

go get github.com/intrepidkarthi/orderbook/pkg/matching@v0.1.0
eng := matching.NewEngine(matching.DefaultConfig("BTC-USD"))

// A resting sell (maker)
ask, _ := types.NewOrder("alice", "BTC-USD", types.SideSell,
    types.OrderTypeLimit, dec("30000"), dec("0.5"),
    types.TIFGoodTillCancel)
eng.Process(ask)

// A crossing buy (taker) — trades against it
bid, _ := types.NewOrder("bob", "BTC-USD", types.SideBuy,
    types.OrderTypeLimit, dec("30000"), dec("0.3"),
    types.TIFGoodTillCancel)
res := eng.Process(bid)

for _, t := range res.Trades {
    fmt.Printf("%s @ %s\n", t.Quantity, t.Price)   // 0.3 @ 30000
}

Runnable examples: go run ./examples/basic, ./examples/signals, ./cmd/obdemo. Full API on pkg.go.dev.
